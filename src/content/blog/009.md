## Introduction

Databases are one of the most important parts of system design. Almost every application needs a way to store, retrieve, and update data reliably. Because of that, system design interviews often test whether you understand database types, scaling strategies, replication, sharding, performance optimization, and tradeoffs like CAP and ACID.

A database is not just a place where data sits. The way data is stored affects how quickly an application can respond, how safely it handles updates, and how well it scales as more users join the system.

## Relational Databases

Relational databases organize data into tables made up of rows and columns. They are useful when the data has a clear structure and relationships between different pieces of information matter. Common examples include PostgreSQL, MySQL, and SQLite.

Relational databases use SQL, or Structured Query Language, to read and modify data. They are especially strong for transactions, complex queries, and data integrity. For example, banking systems, order systems, and user account systems often depend on relational databases because correctness is extremely important.

One major reason relational databases are trusted is that they usually follow ACID properties.

## ACID Properties

ACID stands for atomicity, consistency, isolation, and durability. These properties describe how a database should handle transactions.

Atomicity means a transaction is all-or-nothing. If one part of the transaction fails, the whole transaction should fail instead of leaving the database in a partially updated state.

Consistency means the database should remain valid before and after a transaction. The data should still follow the rules defined by the database, such as constraints, relationships, and required fields.

Isolation means multiple transactions should not interfere with each other in a way that creates incorrect results. Even if many users are using the system at the same time, each transaction should behave as if it is being handled safely.

Durability means that once a transaction is committed, the data should remain saved, even if the system crashes afterward.

Together, these properties help make relational databases reliable for systems where correctness matters.

## NoSQL Databases

NoSQL databases are designed with more flexibility than traditional relational databases. They are often used when the data is less structured, when the system needs to scale quickly, or when the application does not require complex SQL queries.

Examples of NoSQL databases include MongoDB, Cassandra, Redis, and Neo4j. Each one is built around a different data model.

Key-value stores, such as Redis, store data using simple key-value pairs. They are fast and often used for caching, sessions, and quick lookups.

Document databases, such as MongoDB, store data in flexible document-like records. This can be useful when the structure of the data changes often or does not fit neatly into tables.

Graph databases, such as Neo4j, are designed around relationships between data. They are useful for systems like social networks, recommendation engines, and anything where connections between entities are important.

NoSQL databases are often described as schema-less, meaning they do not require the same fixed table structure as relational databases. This flexibility can make development faster, but it also means the application may need to handle more responsibility for data consistency.

## In-Memory Databases

In-memory databases store data mainly in RAM instead of on disk. Because memory is much faster than disk storage, these databases can return data very quickly.

Redis and Memcached are common examples. They are often used for caching, storing user sessions, rate limiting, and holding temporary data that needs to be accessed quickly.

The tradeoff is that memory is more expensive and usually more limited than disk storage. In-memory databases are usually used alongside a main database rather than as the only long-term storage system.

## Vertical and Horizontal Scaling

As an application grows, the database may need to handle more users, more data, and more requests. There are two major ways to scale a database: vertically and horizontally.

Vertical scaling, also called scaling up, means improving a single database server. This can involve adding more CPU power, more RAM, faster storage, or better networking. Vertical scaling is often simpler because the system still uses one main machine, but there is a limit to how powerful one server can become.

Horizontal scaling, also called scaling out, means adding more machines to the system. Instead of relying on one powerful database server, the workload is distributed across multiple servers. This approach can support much larger systems, but it also introduces more complexity.

Two common techniques used in horizontal scaling are sharding and replication.

## Sharding

Sharding means splitting a large dataset into smaller parts called shards. Each shard is stored on a different server. Instead of every server storing the entire database, each server is responsible for only part of the data.

For example, a system might store users with IDs 1 through 1,000,000 on one shard and users with IDs 1,000,001 through 2,000,000 on another. This reduces the load on each individual database server.

There are different ways to shard data.

Range-based sharding splits data based on ranges of a key, such as user IDs or timestamps. Directory-based sharding uses a lookup service to figure out which shard contains the data. Geographic sharding stores data based on location, which can reduce latency for users in different regions.

Sharding can improve scalability, but it also makes the system more complex. The application needs a reliable way to route requests to the correct shard, rebalance data, and handle failures.

## Replication

Replication means keeping copies of data on multiple database servers. This helps improve availability and can also improve read performance.

In primary-replica replication, one primary database handles writes, while one or more replicas handle reads. This is useful when a system has far more read traffic than write traffic. For example, many users might be viewing product pages, but fewer users are updating product information.

Multi-primary replication allows multiple databases to accept writes. This can improve availability, but it is harder to manage because the system must deal with conflicts when the same data is updated in different places.

Replication helps make systems more reliable, but it also introduces the possibility of replication lag. This means a replica may temporarily be behind the primary database, so users might read slightly outdated data.

## Improving Database Performance

Scaling the database is one challenge, but making queries fast is another. Several techniques can help improve database performance.

Caching stores frequently accessed data in a faster storage layer, often using Redis or Memcached. This reduces the number of repeated database queries and can significantly improve response times.

Indexing helps the database find data faster. An index is similar to the index of a book: instead of scanning every row, the database can use the index to jump closer to the data it needs. Indexes are especially useful on columns that are searched, filtered, or sorted often. However, indexes also take up storage and can slow down writes, so they should be used thoughtfully.

Query optimization means improving how a query is written or executed. This can include reducing unnecessary joins, avoiding repeated work, selecting only the needed columns, and using tools like EXPLAIN to understand how the database is processing a query.

Good performance usually comes from a combination of better data modeling, smart indexing, caching, and careful query design.

## CAP Theorem and Tradeoffs

When working with distributed databases, the CAP theorem becomes important. CAP stands for consistency, availability, and partition tolerance.

Consistency means every read receives the most recent correct data. Availability means the system continues responding to requests. Partition tolerance means the system keeps working even when communication between parts of the system fails.

In a distributed system, network failures can happen. Because of that, systems often need to make tradeoffs between consistency and availability during a partition. Some systems prioritize always returning a response, even if the data may be slightly stale. Others prioritize correctness and may reject or delay requests until the data can be safely verified.

There is no single best choice. The right tradeoff depends on what the system needs. A banking system usually cares more about strong consistency, while a social media feed may accept slightly stale data in exchange for higher availability.

## Final Thoughts

Databases are a core part of system design because they affect correctness, speed, scalability, and reliability. Relational databases are strong for structured data and transactions, while NoSQL databases provide flexibility and scalability for different types of workloads.

As systems grow, techniques like vertical scaling, horizontal scaling, sharding, replication, caching, indexing, and query optimization become more important. Each technique solves a problem, but each one also introduces tradeoffs.

A good system design is not about choosing the most advanced database or the most complicated architecture. It is about understanding the requirements of the system and choosing the database approach that fits those requirements best.
